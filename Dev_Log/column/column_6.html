TelpCamp 開発コラム

コラム6: アプリケーションの最後の砦 - サーバーサイドバリデーション

TelpCampでは、Bootstrapの機能を使って「クライアントサイドバリデーション」を実装し、ユーザーが空のフォームを送信できないようにしました。しかし、これはあくまでユーザー体験を向上させるためのもので、セキュリティ対策としては不十分です。悪意のあるユーザーは、このチェックを簡単に迂回して不正なデータを送信できてしまいます。

そこで不可欠となるのが「サーバーサイドバリデーション」です。これは、データがデータベースに保存される直前に、サーバー側でそのデータが正しい形式かどうかを厳密にチェックする仕組みです。TelpCampでは、この目的のためにJoiという非常に人気のあるライブラリを使用しています。

実装の解説

1. ルールブックの作成 (schemas.js)

まず、どのようなデータを受け入れるかの「ルールブック」をschemas.jsに定義しました。例えば、キャンプ場の価格は必須で、0以上の数字でなければならない、といったルールを記述します。 

// schemas.js
const Joi = require('joi');

module.exports.campgroundSchema = Joi.object({
    campground: Joi.object({
        name: Joi.string().required(),
        price: Joi.number().required().min(0),
        // ...
    }).required()
});

2. バリデーションミドルウェアの作成

次に、定義したルールを使って実際にデータを検証するミドルウェアを作成しました。例えば、キャンプ場用のvalidateCampgroundミドルウェアは以下のようになっています。 

// routes/campgrounds.js
const validateCampground = (req, res, next) => {
    const { error } = campgroundSchema.validate(req.body);
    if (error) {
        const msg = error.details.map(el => el.message).join(',');
        throw new ExpressError(msg, 400);
    } else {
        next();
    }
}

3. ルートへの適用

最後に、このミドルウェアをデータを作成・更新するルートに組み込みます。これにより、不正なデータがルートハンドラのロジックに到達する前にブロックされます。 

router.post('/', isLoggedIn, validateCampground, wrapAsync(async (req, res) => { ... }));
router.put('/:id', isLoggedIn, isAuthor, validateCampground, wrapAsync(async (req, res) => { ... }));

💡 防御は多層的に (Defense in Depth)

クライアントサイドとサーバーサイドの両方でバリデーションを行うことは、「多層防御」というセキュリティの基本原則です。クライアントサイドはUXを、サーバーサイドはデータの整合性と安全性を担保します。この両方を実装することで、TelpCampはより堅牢で信頼性の高いアプリケーションになっています。
