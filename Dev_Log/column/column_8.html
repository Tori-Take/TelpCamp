TelpCamp 開発コラム

コラム8: 意図したエラーを的確に - カスタムエラーハンドリング

アプリケーション開発では、単にエラーをキャッチするだけでなく、「どのようなエラー」が「なぜ」発生したのかを明確にすることが重要です。例えば、「ページが見つからない(404)」と「サーバー内部のエラー(500)」では、ユーザーに伝えるべき内容も、開発者が対処すべき内容も全く異なります。

この目的のために、TelpCampではExpressErrorというカスタムエラークラスを導入しました。これにより、エラーメッセージとHTTPステータスコードをセットにして、より意味のあるエラーを生成できるようになります。

実装の解説

1. カスタムエラークラスの作成 (utils/ExpressError.js)

JavaScriptの標準Errorクラスを継承して、messageに加えてstatusCodeプロパティを持つ、独自のExpressErrorクラスを作成しました。 

// utils/ExpressError.js
class ExpressError extends Error {
    constructor(message, statusCode) {
        super();
        this.message = message;
        this.statusCode = statusCode;
    }
}

module.exports = ExpressError;

2. 中央集権的なエラーハンドラ (index.js)

アプリケーションの最後に、すべてのエラーを集約して処理する「エラーハンドリングミドルウェア」を配置しました。このミドルウェアは、next(err)によって渡されたエラーオブジェクトを受け取り、そのstatusCodeとmessageを使って適切なレスポンスを返します。 

// index.js
app.use((err, req, res, next) => {
    const { statusCode = 500, message = '問題が起きました' } = err;
    // 将来的にはエラー表示用の専用ページをレンダリングする
    res.status(statusCode).send(message);
});

3. カスタムエラーの使用例

例えば、どのルートにも一致しなかった場合に404エラーを生成するルートは、このカスタムエラークラスを使って以下のように簡潔に記述できます。 

// index.js
app.use((req, res, next) => {
    next(new ExpressError('ページが見つかりませんでした。', 404));
});

💡 エラー処理の重要性

適切なエラーハンドリングは、堅牢なアプリケーションの根幹です。catchAsyncでエラーを確実に捕捉し、ExpressErrorでエラーに意味を与え、中央のエラーハンドラで一貫した方法でユーザーに応答する。この一連の流れが、TelpCampの安定性と保守性を支えています。
