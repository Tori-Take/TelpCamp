Step 18: サーバーサイドバリデーションの実装 (Joi)

前回までのクライアントサイドバリデーションは、ユーザー体験を向上させるための「第一の防衛線」でした。しかし、このチェックは簡単に回避できてしまいます。 今回は、アプリケーションの堅牢性とデータの整合性を確保するための「第二の、そして最も重要な防衛線」であるサーバーサイドバリデーションを、キャンプ場の登録・更新機能に実装します。

この実装には、JavaScriptのオブジェクトスキーマを検証するための強力なライブラリであるjoiを使用します。

学習の目的

  • クライアントサイドバリデーションだけでは不十分な理由を理解する。
  • joiライブラリを使い、データの検証ルール（スキーマ）を定義する方法を学ぶ。
  • Expressのミドルウェアとしてバリデーションロジックを実装し、特定のルートに適用する方法を習得する。
  • バリデーションエラーが発生した場合に、適切なエラーメッセージとステータスコードを返す方法を実践する。

実装の解説

キャンプ場の新規登録(Create)と更新(Update)の両方のルートで、データがデータベースに保存される前に、その内容が正しい形式であるかを厳密にチェックする仕組みを導入します。

  1. 1. joiのインストール

    まず、プロジェクトにjoiパッケージをインストールします。 

    npm install joi

  2. 2. バリデーションスキーマの作成 (schemas.js)

    次に、どのようなデータを受け入れるかの「ルールブック」となるバリデーションスキーマを定義します。プロジェクトのルートにschemas.jsという新しいファイルを作成し、キャンプ場用のスキーマを記述します。 

    const Joi = require('joi');

module.exports.campgroundSchema = Joi.object({
    campground: Joi.object({
        name: Joi.string().required(),
        location: Joi.string().required(),
        image: Joi.string().required(),
        price: Joi.number().required().min(0),
        description: Joi.string().required()
    }).required()
});

    💡 Joiスキーマの読み解き方

    このコードは「受け取るデータ(req.body)は、campgroundというキーを持つオブジェクトでなければならない。そして、そのcampgroundオブジェクトは、以下のルールに従うキーを持つ必要がある」と定義しています。

    • name: 必須(required)の文字列(string)でなければならない。
    • price: 必須の数値(number)で、かつ0以上(min(0))でなければならない。

  3. 3. バリデーションミドルウェアの作成と適用 (index.js)

    作成したスキーマを使って、実際にリクエストのデータを検証するミドルウェアをindex.jsに作成し、それを新規登録(POST)と更新(PUT)のルートに適用します。 

    // Joiスキーマを読み込み
const { campgroundSchema } = require('./schemas.js');

// キャンプ場用のバリデーションミドルウェア
const validateCampground = (req, res, next) => {
    const { error } = campgroundSchema.validate(req.body);
    if (error) {
        const msg = error.details.map(el => el.message).join(',');
        throw new ExpressError(msg, 400);
    } else {
        next();
    }
}

// 新規登録(Create)ルートにミドルウェアを適用
app.post('/campgrounds', validateCampground, wrapAsync(async (req, res) => { ... }));

// 更新(Update)ルートにミドルウェアを適用
app.put('/campgrounds/:id', validateCampground, wrapAsync(async (req, res) => { ... }));

    💡 ミドルウェアによる関心の分離

    バリデーションのロジックをvalidateCampgroundという独立したミドルウェアに切り出すことで、ルートハンドラは「データが正しい」という前提で、データベース操作などの本来の仕事に集中できます。

    もしバリデーションに失敗した場合、ミドルウェアはExpressErrorをスローします。このエラーはwrapAsyncによってキャッチされ、最終的に中央エラーハンドラに渡されて、ユーザーにエラーメッセージが表示される流れになっています。
