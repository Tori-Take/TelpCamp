Step 20: レビュー投稿フォームの作成とバリデーション実装

前回までのモデル設計に続き、今回はユーザーが実際にレビューを投稿するためのUI（フォーム）と、そのデータを安全に受け取るためのロジックを実装します。 ここでも、これまでの学習で身につけたフォーム作成、ルーティング、そしてクライアントサイド・サーバーサイド両面からのバリデーションの知識を総動員します。

学習の目的

  • 既存のページ（詳細ページ）に、新しい機能（レビュー投稿）のフォームを追加する方法を実践する。
  • ネストされたリソース（キャンプ場に属するレビュー）を処理するためのPOSTルートを作成する方法を学ぶ。
  • 新しいReviewドキュメントを作成し、親であるCampgroundドキュメントと正しく関連付けるロジックを実装する。
  • レビュー投稿フォームに、クライアントサイドとサーバーサイドの両方のバリデーションを適用し、アプリケーションの堅牢性を高める。

実装の解説

キャンプ場の詳細ページにレビュー投稿フォームを設置し、投稿されたデータを検証した上でデータベースに保存する、という一連の流れを実装します。

  1. 1. レビュー投稿フォームの追加 (views/campgrounds/show.ejs)

    まず、キャンプ場の詳細ページの下部に、レビューを入力・送信するためのフォームを追加します。このフォームには、Bootstrapを使ったクライアントサイドバリデーションも適用します。 

    <h2>レビューを投稿する</h2>
<form action="/campgrounds/<%= campground._id %>/reviews" method="POST" class="needs-validation" novalidate>
    <div class="mb-3">
        <label class="form-label" for="rating">評価</label>
        <input class="form-range" type="range" min="1" max="5" name="review[rating]" id="rating">
    </div>
    <div class="mb-3">
        <label class="form-label" for="body">コメント</label>
        <textarea class="form-control" name="review[body]" id="body" cols="30" rows="3" required></textarea>
        <div class="invalid-feedback">
            コメントを入力してください。
        </div>
    </div>
    <button class="btn btn-success">投稿</button>
</form>

    💡 ネストされたリソースのURL設計

    フォームのaction属性を/campgrounds/<%= campground._id %>/reviewsと設定しました。 これは「特定のキャンプ場（/campgrounds/:id）に属する、レビュー（/reviews）というリソースを作成する」という意図をURLで明確に表現する、RESTfulな設計の良い実践例です。

  2. 2. レビュー作成ルートの実装 (index.js)

    フォームから送信されたデータを受け取り、データベースに保存するためのPOSTルートをindex.jsに追加します。 

    // レビュー作成処理のPOSTリクエスト
app.post('/campgrounds/:id/reviews', validateReview, wrapAsync(async (req, res) => {
    // 1. 対象のキャンプ場を検索
    const campground = await Campground.findById(req.params.id);
    // 2. フォームデータで新しいレビューインスタンスを作成
    const review = new Review(req.body.review);
    // 3. キャンプ場のreviews配列に新しいレビューを追加
    campground.reviews.push(review);
    // 4. 両方の変更をデータベースに保存
    await review.save();
    await campground.save();
    // 5. 元のキャンプ場詳細ページにリダイレクト
    res.redirect(`/campgrounds/${campground._id}`);
}));

    💡 farmstandプロジェクトの知識の応用

    このロジックは、farmstandプロジェクトで「農場に商品を追加する」処理と全く同じです。 親ドキュメント（Campground）と子ドキュメント（Review）の両方をインスタンス化し、互いを関連付けた後、両方をsave()しています。 一度学んだパターンを、異なる機能で再利用できる良い例です。

  3. 3. レビューのサーバーサイドバリデーション

    最後に、クライアント側のチェックを回避された場合に備え、サーバー側でもjoiを使って厳密なデータ検証を行います。

    schemas.js

    module.exports.reviewSchema = Joi.object({
    review: Joi.object({
        rating: Joi.number().required().min(1).max(5),
        body: Joi.string().required()
    }).required()
});

    index.js

    // バリデーションミドルウェア
const validateReview = (req, res, next) => {
    const { error } = reviewSchema.validate(req.body);
    if (error) {
        const msg = error.details.map(el => el.message).join(',');
        throw new ExpressError(msg, 400);
    } else {
        next();
    }
}

// ルートにミドルウェアを適用
app.post('/campgrounds/:id/reviews', validateReview, wrapAsync(async (req, res) => { ... }));

    キャンプ場のバリデーションと同様に、レビュー用のスキーマとミドルウェアを作成し、それをPOSTルートに適用しました。 これにより、不正なデータがデータベースに保存されるのを防ぎ、アプリケーションの信頼性を高めています。
