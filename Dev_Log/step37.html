TelpCamp プロジェクト作成の記録

Step 37: キャンプ場の認可（所有権）

アプリケーションのセキュリティをさらに強化するため、キャンプ場の編集・削除といった破壊的な操作を、そのキャンプ場を登録した本人（所有者）のみに許可する認可機能を実装しました。

学習の目的

  • 所有者ベースの認可ミドルウェアを作成する方法を学ぶ。
  • Mongooseの.equals()メソッドを使い、ObjectIdを正しく比較する方法を習得する。
  • クライアントサイド（ビュー）とサーバーサイド（ルート）の両方でアクセス制御を行う、二段構えのセキュリティ実装を理解する。

実装の解説

所有者かどうかをチェックするミドルウェアを作成し、それをルートとビューの両方に適用しました。

1. 認可ミドルウェアの作成 (middleware.js)

キャンプ場の所有者と現在ログインしているユーザーが一致するかを検証するisAuthorミドルウェアを作成しました。 

// middleware.js
const Campground = require('./models/campground');

module.exports.isAuthor = async (req, res, next) => {
    const { id } = req.params;
    const campground = await Campground.findById(id);
    // キャンプ場の所有者と、現在ログインしているユーザーが一致しない場合
    if (!campground.author.equals(req.user._id)) {
        req.flash('error', 'その操作を行う権限がありません。');
        return res.redirect(`/campgrounds/${id}`);
    }
    next();
}

💡 ObjectIdの比較

campground.authorとreq.user._idはどちらもMongooseのObjectIdオブジェクトです。これらをJavaScriptの厳密等価演算子（===）で比較すると、参照先が異なるため常にfalseになります。ObjectIdの値を比較するには、Mongooseが提供する.equals()メソッドを使用する必要があります。

2. サーバーサイドでの認可 (routes/campgrounds.js)

作成したisAuthorミドルウェアを、編集フォームの表示（GET）、更新（PUT）、削除（DELETE）の各ルートに適用しました。これにより、URLを直接操作するなどの不正なリクエストをサーバー側でブロックします。 

// routes/campgrounds.js
const { isLoggedIn, isAuthor } = require('../middleware');

router.get('/:id/edit', isLoggedIn, isAuthor, ...);
router.put('/:id', isLoggedIn, isAuthor, ...);
router.delete('/:id', isLoggedIn, isAuthor, ...);

3. クライアントサイドでの認可 (views/campgrounds/show.ejs)

ユーザー体験を向上させるため、そもそも権限のないユーザーには「編集」「削除」ボタンが表示されないように、ビューファイルを修正しました。 

<% if(currentUser && campground.author.equals(currentUser._id)) { %>
    <a href="/campgrounds/<%= campground._id %>/edit" class="card-link btn btn-info">編集</a>
    <form ...>
        <button class="btn btn-danger">削除</button>
    </form>
<% } %>

このクライアント側とサーバー側の二重のチェックにより、堅牢で安全な認可システムが完成しました。
